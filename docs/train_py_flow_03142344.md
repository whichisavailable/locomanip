# `train.py` Training Flow 03142344

这份文档专门解释 `scripts/reinforcement_learning/rsl_rl/train.py` 在做什么，以及其中最容易让人困惑的一点：

为什么脚本先解析命令行，再用 Hydra 加载配置，最后又用命令行去覆盖一部分配置。

这不是重复设计，而是分层设计。

---

## 1. 先给出最短结论

`train.py` 不是“算法实现文件”，而是“训练启动与装配文件”。

它负责的事情可以概括成五步：

1. 接收外部输入
2. 启动模拟器和运行时环境
3. 根据任务名加载环境配置和算法配置
4. 用本次运行的参数修正这些配置
5. 创建环境和 runner，正式开始训练

所以它的核心作用不是定义 PPO，而是把“任务系统”和“训练器”接起来。

---

## 2. `train.py` 可以分成三层

理解这个文件时，最有效的办法不是逐行看，而是先分层。

### 第一层：启动层

这一层负责脚本刚开始运行时必须完成的事情：

- 定义命令行参数
- 解析命令行
- 启动 Isaac Sim
- 检查依赖版本

这一层处理的是“程序怎么启动起来”。

### 第二层：配置装配层

这一层负责把任务和算法配置对象准备好：

- 根据 `--task` 找到对应任务
- 根据 `--agent` 找到对应算法配置入口
- 通过 `@hydra_task_config(...)` 把 `env_cfg` 和 `agent_cfg` 实例化出来

这一层处理的是“本次训练到底要训练什么、用什么训练器”。

### 第三层：运行层

这一层发生在 `main(...)` 里：

- 修正配置
- 创建环境
- 创建 runner
- 加载 checkpoint
- 调用 `learn()`

这一层处理的是“正式跑训练”。

---

## 3. 参数其实不是一类，而是三类

你之所以会觉得“既然 CLI 和 Hydra 分工了，为什么又覆盖”，主要是因为脚本里的参数表面看起来都混在一起。

实际上，这里至少有三类参数。

### 第一类：启动参数

这类参数只和“脚本本次怎么启动”有关，不属于任务或算法本身的长期定义。

例如：

- `--video`
- `--video_length`
- `--video_interval`
- `--distributed`
- `--device`
- `--ray-proc-id`

这些参数的特点是：

- 和本次运行方式强相关
- 经常临时变化
- 没必要写进任务默认配置里

这类参数天然适合放在 CLI。

### 第二类：基础配置参数

这类参数属于任务或算法定义本身。

例如：

- 环境的 observation、reward、action
- PPO 网络结构
- 学习率
- 默认训练轮数
- 默认实验名

这些参数的特点是：

- 有明显层级结构
- 更适合作为任务默认配置保存
- 需要继承、复用和版本化

这类参数天然适合放在 Hydra/配置对象里。

### 第三类：运行时覆盖参数

这类参数很关键。它们逻辑上属于配置对象，但在实际训练中又经常需要临时改。

例如：

- `--seed`
- `--num_envs`
- `--max_iterations`
- `--resume`
- `--load_run`
- `--checkpoint`
- `--logger`
- `--run_name`

这类参数如果只放在配置里，会导致一个很烦的结果：
每次只想做一次短实验，都得改配置类或写 Hydra override。

所以脚本给了它们一个更方便的入口：CLI。

这就是为什么会出现“Hydra 先加载，CLI 再覆盖”。

因为这里不是两套系统各管完全不相交的参数，而是：

- Hydra 提供结构化默认值
- CLI 提供本次运行的快捷覆盖入口

---

## 4. 正确理解方式不是“分家”，而是“分层”

你之前的理解是：

- CLI 负责一部分参数
- Hydra 负责另一部分参数

这个理解方向对，但还不够准确。

更准确的说法是：

- CLI 和 Hydra 不是完全分家
- 它们是有主次关系和优先级关系的两层

更接近实际的模型是：

1. 先有任务默认配置
2. 再有算法默认配置
3. Hydra 把这些默认配置对象加载出来
4. CLI 对本次运行需要临时改的字段做覆盖
5. 脚本再根据运行环境做最后修正

所以这里不是“谁管谁”，而是“谁先提供基线，谁后提供修正”。

---

## 5. 一条真正的配置优先级链

如果把 `train.py` 里的配置来源按优先级从低到高排，可以写成：

1. 任务注册决定加载哪个 `env_cfg` 和 `agent_cfg`
2. `env_cfg` / `agent_cfg` 类本身提供默认值
3. Hydra 把这些默认值实例化成对象
4. CLI 覆盖本次运行想改的字段
5. 脚本根据硬件和运行模式再做派生修正

最后一层“派生修正”也很重要，因为它说明脚本并不是只做参数搬运。

例如在 `main(...)` 里还会发生这些事情：

- 如果用户传了 `--num_envs`，覆盖 `env_cfg.scene.num_envs`
- 如果用户传了 `--max_iterations`，覆盖 `agent_cfg.max_iterations`
- 如果 GPU 显存太小且没显式给 `--num_envs`，自动缩小环境数
- 如果分布式训练，自动把设备改成当前 rank 的 GPU
- 如果分布式训练，自动给不同 rank 加 seed 偏移
- 自动生成 `log_dir`

这类逻辑已经不是“静态配置”，而是“运行期推导”。

## 6. `parse_known_args()` 为什么很关键

脚本没有直接用 `parse_args()`，而是用了：

`args_cli, hydra_args = parser.parse_known_args()`

这一步非常关键，因为它把参数分成了两部分：

- `args_cli`
  - 训练脚本自己认识的参数
- `hydra_args`
  - 训练脚本不直接处理，但要留给 Hydra 的参数

这说明脚本的思路是：

1. 先把“我自己必须先处理的参数”拿出来
2. 其余的不要报错，留给 Hydra

为什么必须先处理？

因为像下面这些事情在 Hydra 装配配置之前就要知道：

- 要不要录视频
- 要不要开相机
- 用什么设备
- 启动 AppLauncher 需要哪些参数

所以这里的顺序不是随便写的，而是启动流程决定的。

---

## 7. 为什么要重写 `sys.argv`

脚本里有这一行：

`sys.argv = [sys.argv[0]] + hydra_args`

这表示：

- 之前已经被 `argparse` 吃掉的那部分 CLI 参数，不再交给 Hydra
- 只把剩下的 `hydra_args` 交给 Hydra 继续处理

这一步的目的很明确：
避免同一批参数被两套解析器重复处理。

所以它的意思不是“把命令行清空”，而是“给 Hydra 留一份它该看到的参数列表”。

---

## 8. 为什么 Hydra 加载完以后还要 CLI 覆盖

这是这份文档最核心的问题。

答案是：

因为 Hydra 在这里的职责是“提供结构化默认配置”，不是“禁止本次运行做快捷修改”。

可以把这一段逻辑理解成：

1. 先通过任务注册和配置类得到一份完整默认配置
2. 再根据这次命令行输入，把少数常用字段临时改掉

最典型的代码在这里：

- `agent_cfg = cli_args.update_rsl_rl_cfg(agent_cfg, args_cli)`
- `env_cfg.scene.num_envs = args_cli.num_envs if args_cli.num_envs is not None else env_cfg.scene.num_envs`
- `agent_cfg.max_iterations = args_cli.max_iterations if args_cli.max_iterations is not None else agent_cfg.max_iterations`

这说明 CLI 覆盖的不是整个配置树，而是少数“高频实验参数”。

为什么要这样做？

因为这些字段虽然属于配置对象，但用户很常在训练命令里改它们：

- 改 seed
- 改迭代轮数
- 改日志名字
- 改 resume 行为
- 改环境并行数

如果每次都要求你去改配置文件，实验效率会很差。

所以真正的分工不是：

- CLI 和 Hydra 各自负责完全不同的参数集合

而是：

- Hydra 提供完整基线
- CLI 为常改项提供快捷入口

---

## 9. 为什么这不算设计冲突

这套设计看起来像“重复定义参数”，但工程上其实很常见，因为它解决的是“默认值”和“本次运行需求”之间的张力。

如果只有 Hydra，没有 CLI 快捷覆盖，会出现这些问题：

- 临时改一个 seed 很麻烦
- 只想跑 1000 轮测试时不方便
- resume/checkpoint 这类运行控制项不够顺手
- 外部调度系统传参不方便

如果只有 CLI，没有结构化配置，会出现这些问题：

- 环境和算法配置会散成一堆平铺参数
- 任务复用困难
- 配置继承困难
- 很难表达复杂 observation/reward/action 结构

所以这套设计不是多余，而是把两种能力叠起来：

- Hydra 负责“配置组织能力”
- CLI 负责“实验操作便利性”

## 10. 哪些参数更像“覆盖”，哪些参数更像“推导”

在 `main(...)` 里，并不是所有写回配置对象的操作都属于同一种性质。

### 第一种：直接覆盖

例如：

- `seed`
- `run_name`
- `resume`
- `checkpoint`
- `max_iterations`
- `num_envs`

这类操作的特点是：

- 用户明确传了值
- 脚本直接替换配置中的对应字段

### 第二种：条件推导

例如：

- 低显存时自动缩小 `num_envs`
- 分布式训练时自动设置 `cuda:{local_rank}`
- 分布式训练时自动调整 seed

这类操作的特点是：

- 不是用户显式定义的配置值
- 是脚本根据运行条件动态推导出来的

这个区分很重要，因为它能帮助你以后设计自己的训练入口。

你应该问两个问题：

1. 这个值是用户要直接控制的吗？
2. 还是脚本根据上下文自动算出来的？

## 11. `train.py` 中各个重要片段的职责

下面按功能拆一下关键代码片段。

### 参数注册

作用：

- 声明脚本支持哪些外部输入
- 包含训练脚本参数、RSL-RL 参数、AppLauncher 参数

### `parse_known_args()`

作用：

- 先拿到训练脚本必须先知道的参数
- 把其余参数保留给 Hydra

### `AppLauncher(args_cli)`

作用：

- 启动 Isaac Sim
- 这一步属于训练真正开始前的运行环境准备

### `@hydra_task_config(args_cli.task, args_cli.agent)`

作用：

- 按任务名和算法入口名加载配置对象

### `cli_args.update_rsl_rl_cfg(agent_cfg, args_cli)`

作用：

- 把常用 CLI 参数覆盖回 agent 配置对象

### `gym.make(args_cli.task, cfg=env_cfg, ...)`

作用：

- 根据任务名和配置对象创建环境实例

### `RslRlVecEnvWrapper(...)`

作用：

- 把 Isaac Lab 环境适配成 RSL-RL 期望的向量化环境接口

### `OnPolicyRunner(...)`

作用：

- 创建真正负责训练循环的执行器

### `runner.learn(...)`

作用：

- 正式开始训练

## 12. 从工程角度看，这种设计有什么价值

这套设计对后续扩展非常重要。

因为它把系统拆成了比较稳定的边界：

- 任务系统
- 配置系统
- 运行时覆盖层
- 算法执行层

这样你以后要做自己的算法时，不需要重写整个项目。

你通常可以保留：

- CLI 启动流程
- Hydra 配置装配
- 任务注册
- 环境创建

然后只替换：

- wrapper 的一部分适配逻辑
- runner / trainer
- 自己的更新循环

这也是为什么理解 `train.py` 的价值不在于“会读一个脚本”，而在于它定义了整个训练系统的边界。

---

## 13. 这部分最值得记住的五句话

1. `train.py` 负责启动和装配，不负责实现 PPO 细节。
2. Hydra 在这里负责加载结构化默认配置。
3. CLI 不只是启动开关，也承担了高频实验参数的快捷覆盖入口。
4. CLI 和 Hydra 不是完全分家，而是分层叠加。
5. 配置最终值不是只来自某一个地方，而是来自“默认配置 + CLI 覆盖 + 运行期推导”。

---

## 14. 一句话版本

`train.py` 的真实逻辑不是“CLI 和 Hydra 各管一半”，而是：

先用 Hydra 组装一份完整配置，再用 CLI 把本次实验最常变的那些字段快捷改掉，最后由脚本根据运行环境做收尾修正。
